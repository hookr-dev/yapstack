# Plan: Expand Parakeet model selection

## Context

Today only **Parakeet TDT v3** is offered (`crates/yapstack-transcription/src/model.rs:88-145`).
Whisper offers four sizes (Tiny / Base / Small / Medium); Parakeet should offer a
comparable selection. The underlying `parakeet-rs 0.3` crate already supports
multiple loader types, and `istupakov` / `onnx-community` host compatible ONNX bundles.

## What's available (researched 2026-04-25)

| Variant | Loader (parakeet-rs) | HF repo | Lang | Streaming | Notes |
|---|---|---|---|---|---|
| **TDT v3** *(current)* | `ParakeetTDT` | `istupakov/parakeet-tdt-0.6b-v3-onnx` | 25 EU langs | No | Multilingual default |
| **TDT v2** | `ParakeetTDT` (same loader) | `istupakov/parakeet-tdt-0.6b-v2-onnx` | English-only | No | Higher EN accuracy than v3 (v3 traded EN WER for multilingual) |
| **CTC 0.6B** | `Parakeet` (different loader) | `onnx-community/parakeet-ctc-0.6b-ONNX` | English-only | No | Smaller/faster decode, native punctuation & capitalization |
| **EOU streaming** | `ParakeetEOU` | `altunenes/parakeet-rs` (realtime_eou_120m-v1-onnx) | English-only | Yes (160 ms chunks) | Native end-of-utterance — could replace VAD-driven chunking later |
| **Nemotron 0.6B** | `Nemotron` | `altunenes/parakeet-rs` (nemotron-speech-streaming-en-0.6b) | English-only | Yes (560 ms chunks) | Cache-aware streaming with punctuation |
| **RNNT 0.6B** | (not in parakeet-rs README) | `istupakov/parakeet-rnnt-0.6b-onnx` | English-only | No | Superseded by TDT — skip |

## Recommendation

Ship in two phases. Phase 1 is the cheap win; phase 2 is a real new code path.

### Phase 1 — TDT v2 (English-only)

~30 min. Same loader as v3, same file layout, same Sortformer pipeline. The
sidecar's `engines/parakeet.rs` doesn't care about variant — it gets a directory
path and calls `ParakeetTDT::from_pretrained`. So the work is purely:
metadata + DTOs + UI labels.

**Touchpoints**

1. `crates/yapstack-transcription/src/model.rs`
   - Add `TdtV2` to `ParakeetVariant` enum
   - Add cases to `dir_name`, `files`, `display_name`, `approximate_size_bytes`
   - Update `all()` to include both
   - **Also fix the wrong v3 file sizes while here** (see "Bug discovered" below)

2. `apps/desktop/src-tauri/src/commands/transcription.rs:63-65, 140-153`
   - Add `TdtV2` to `ParakeetVariantDto`
   - Add cases to both `From` impls

3. `apps/desktop/src/components/StatusPopover.tsx:240-245`
   - Add label mapping `"TdtV2" → "TDT v2"`

4. `apps/desktop/src/components/settings/ParakeetModelSection.tsx:122`
   - Update "Recommended" badge logic — keep `TdtV3` as recommended (multilingual default).
     Optionally add a second badge `English-only` on `TdtV2`.

5. Run `pnpm check` — regenerates `apps/desktop/src/lib/types.ts` (auto-generated by tauri-specta).

**No DB migration needed.** Default setting is `TdtV3`; the appStore migration
at `apps/desktop/src/stores/appStore.ts:2054-2055` already defaults missing values to `TdtV3`.

**File list for v2** (verified on HF, 2026-04-25):
- `encoder-model.onnx` — 41.8 MB
- `encoder-model.onnx.data` — 2.44 GB (external initializer, same as v3)
- `decoder_joint-model.onnx` — 35.8 MB
- `vocab.txt` — 9.38 KB
- Total: ~2.5 GB

URL prefix: `https://huggingface.co/istupakov/parakeet-tdt-0.6b-v2-onnx/resolve/main/`

### Caveat (engine catalogue)

`crates/yapstack-common/src/engines.rs:51-66` declares Parakeet's languages as the
v3 set (25 langs). If a user selects TDT v2 + a non-English language, validation
passes but the model will produce poor output. Two options:

- **Quick:** leave catalogue alone; v2 is documented as English-only; users get
  bad transcription if they mismatch. Same UX as the omitted Whisper `.en`
  variants.
- **Right:** make `EngineDescriptor` per-variant, or add a `variant.languages()`
  method and intersect with engine languages in the language picker.

Defer the right fix to Phase 2 unless we get user reports.

### Bug discovered during research

The v3 file sizes in `model.rs` are off by ~4× and the display name is wrong:

| Field | Current | Actual |
|---|---|---|
| `encoder-model.onnx` | 1.7 MB | 41.8 MB |
| `encoder-model.onnx.data` | 560 MB | 2.44 GB |
| `decoder_joint-model.onnx` | 34 MB | 72.5 MB |
| `vocab.txt` | 32 KB | 93.9 KB |
| `display_name` | "Parakeet TDT v3 (~600 MB)" | actual ~2.6 GB |

These sizes are used as `fallback_size` in `download_file()` for progress
reporting when HF doesn't return Content-Length. The display name shown in the
model picker is also wrong by ~4×. Worth fixing in the same PR as v2 — would
look weird if v2 says "~2.5 GB" next to v3 saying "~600 MB" when they're nearly
identical on disk.

### Phase 2 — CTC (English-only "fast tier") — optional later

Roughly half a day. CTC needs the `Parakeet` loader (not `ParakeetTDT`), which
means `engines/parakeet.rs` becomes a dispatch over loader type. Concretely:

- The `model: Option<ParakeetTDT>` field becomes an enum like
  `enum LoadedParakeet { Tdt(ParakeetTDT), Ctc(Parakeet) }`.
- `load_model` picks the loader based on a new `--parakeet-variant` CLI flag
  on the sidecar (or via inspection of the model directory contents).
- `transcribe()` dispatches through the `Transcriber` trait, which both types
  implement, so most of the call site stays the same.
- Sortformer integration unchanged — runs on the same audio regardless.

Worth it if users ask for a faster / smaller-footprint Parakeet option.

### Phase 3 — Streaming variants (EOU / Nemotron) — defer indefinitely

These have fundamentally different APIs (chunk-by-chunk `transcribe(chunk, false)`
or `transcribe_chunk(chunk)` returning incremental text). They'd conflict with
the current VAD-driven `live_transcription.rs` loop, which extracts utterances
between speech/silence transitions and sends complete chunks to the sidecar.

A streaming variant would only pay off as a **separate low-latency feature**
with its own controller — not as a drop-in alternative inside the existing live
transcription pipeline. Not worth the complexity until users ask for sub-second
partial-result latency.
