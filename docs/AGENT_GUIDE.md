# Agent guide

Practical navigation for AI coding agents. Pairs with [`AGENTS.md`](../AGENTS.md) (canonical rules) and [`docs/INDEX.md`](INDEX.md) (router).

## Read this in order on a cold start

1. [`AGENTS.md`](../AGENTS.md) — your operating rules.
2. [`docs/INDEX.md`](INDEX.md) — find the right doc.
3. [`docs/GLOSSARY.md`](GLOSSARY.md) — domain terms before you grep.
4. [`docs/ARCHITECTURE.md`](ARCHITECTURE.md) — only the section relevant to your task. Don't load the whole file unless you're working cross-cutting.

Skip everything else until a task forces it.

## Common task recipes

### Adding a Tauri command

1. Define DTO structs in `apps/desktop/src-tauri/src/commands/<area>.rs`. Derive `serde::{Serialize, Deserialize}` and `specta::Type` (the latter generates the TS binding).
2. Implement the `#[tauri::command]` `#[specta::specta]` async function. Return `Result<T, CommandError>`.
3. Register in `tauri::generate_handler!` in `apps/desktop/src-tauri/src/lib.rs`.
4. Run `pnpm tauri dev` once to regenerate `apps/desktop/src/lib/tauri.ts`.
5. Frontend calls via `commands.<commandName>()`.

### Adding a sidecar IPC message

1. Extend `SidecarRequest` and/or `SidecarResponse` in `crates/yapstack-common/src/types.rs`.
2. Wire the dispatcher branch in `crates/yapstack-transcription-sidecar/src/main.rs`.
3. Implement on the `TranscriptionBackend` trait if engine-specific.
4. Surface in `TranscriptionClient` (`crates/yapstack-transcription/src/client.rs`).
5. **Critical**: forward + backward compatibility across mismatched sidecar/host versions. Bumping the protocol requires an ADR.

### Adding a database column

1. Update `apps/desktop/src-tauri/src/db.rs` migration list — append a new versioned migration. Don't edit existing migrations.
2. If the migration must run before `tauri-plugin-sql` initialization, add it to `db::ensure_runtime_schema()`.
3. Update the frontend type in `apps/desktop/src/lib/db.ts`.
4. Add a test that exercises the new column.
5. **Schema migrations are an Ask-first operation per AGENTS.md.**

### Adding a Zustand store field with persistence

1. Add the field to the state type in `apps/desktop/src/stores/appStore.ts`.
2. Bump the persist `version` and add a migration step in the `migrate` callback.
3. Test the migration by clearing local storage and confirming the default value is applied.
4. Document the field in [`docs/FRONTEND.md`](FRONTEND.md) if it affects UX.

### Adding an AI chat tool

1. Add a `registerTool({...})` call in `apps/desktop/src/lib/ai-tools.ts`. Define `name`, `description`, JSON Schema for parameters, and an `execute` function.
2. The chat loop will pick it up automatically (multi-turn execution loop already wired).
3. If the tool produces user-visible state changes, ensure `ToolExecutionBlock` renders something useful for it.
4. Add a test in `ai-tools.test.ts`.
5. See [`docs/AI_CONTEXT.md`](AI_CONTEXT.md) for the tool registry contract.

### Editing transcription pipeline behaviour

1. Read the live transcription section of [`docs/ARCHITECTURE.md`](ARCHITECTURE.md) end-to-end before touching `commands/live_transcription.rs`. There's tight coupling between cursors, VAD state machines, prompt context, and stream health monitoring.
2. The `live_pressure` and `live_engine_loaded` log markers are how you observe runtime behaviour. Add new markers via `tracing::info!(marker = "...", ...)`.
3. Backfill must continue to work concurrently with live; don't break the `origin: "backfill"` flow.
4. Test on a real recording session before declaring done. UI tests don't catch pipeline regressions.

## Debugging tips

- **Sidecar crashes silently** → check stderr; the desktop app captures it into `Settings → General → Logs`.
- **TS binding out of sync** → run `pnpm tauri dev` once; specta regenerates on a successful build.
- **Migration didn't apply** → check `tauri-plugin-sql` migration version table in the SQLite file; you may need to bump and re-run.
- **Auto-updater not firing** → updater hits `releases/latest/download/latest.json`. If the release is marked `prerelease: true`, GitHub's "latest" skips it.
- **WebGPU/CoreML not engaging** → set `YAPSTACK_PARAKEET_ACCEL=cpu|coreml|webgpu` to override. Check the `live_accel_choice` log marker for what actually loaded.

## What good agent output looks like here

- Reads the **smallest** relevant slice of code to make a confident change.
- Runs `pnpm check` (or the targeted subset) before claiming done.
- Updates `CHANGELOG.md` for user-visible changes.
- Adds or updates tests for behaviour changes.
- Writes a tight commit subject + a *why* in the body when non-obvious.
- Uses the `Co-Authored-By:` trailer.
- Stops and asks when about to touch a schema migration, IPC protocol, or anything in the AGENTS.md "Ask first" list.

## What bad agent output looks like

- Loading the whole codebase before starting.
- Writing comments that restate code (`// increment counter`).
- Adding "future-proofing" abstractions for problems we don't have.
- Skipping tests.
- Editing `apps/desktop/src/lib/tauri.ts` by hand.
- Force-pushing without explicit per-action user confirmation.
- Committing `.env`, model files, or anything else `.gitignore` would have caught.
